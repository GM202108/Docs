Ofrece una amplia variedad de [componentes](components/component-catalog.mdx) integrados para realizar diversas tareas, como la serialización de formatos de datos, la invocación de puntos finales REST o la interacción con plataformas SaaS comúnmente utilizadas como AWS, Slack o Dropbox. La mayoría de tus integraciones probablemente se pueden desarrollar utilizando estos componentes estándar.

Para las partes de tus integraciones que son específicas de tu industria o producto, proporcionamos la capacidad de ejecutar código personalizado como parte de tu integración.
Puedes implementar tu propio código en forma de componentes personalizados reutilizables o pasos de código corto [code](components/code.mdx).

## Componentes personalizados

Los componentes personalizados son porciones reutilizables de código relacionado que pueden ser utilizadas por varias de tus integraciones.
Un componente personalizado se desarrolla como un paquete de [node](https://nodejs.org/) y está compuesto por una o más **acciones** que se pueden agregar a una integración.

Por ejemplo, Progix Software podría admitir un formato de datos propietario, `.rocket`, para planos de cohetes.

Progix podría crear un componente personalizado con dos acciones:

1. Serializar un `.rocket` archivo a JSON
1. Deserializar un JSON archivo a `.rocket`

Para crear un componente personalizado, te recomendamos seguir el tutorial [Your First Custom Component](quickstarts/first-custom-component.mdx) para observar cómo funciona el componente [Slack](components/slack.mdx).
Luego, consulta [Writing Custom Components](custom-components/writing-custom-components.mdx) para escribir tu propio componente personalizado.

Para temas avanzados, como el manejo de conexiones o datos binarios, lee [Advanced Custom Components](quickstarts/building-the-s3-component.mdx) para ver cómo el componente incorporado de [Amazon S3](components/aws-s3.mdx) aborda conceptos avanzados de componentes.

## El componente de código

El componente [code](components/code.mdx) se utiliza para código propietario más pequeño y específico de la integración.
Si un problema no se puede resolver con componentes integrados, pero se puede resolver con unas pocas líneas de JavaScript, el componente de código es una buena elección.

Para ver el uso y algunos casos de uso del componente de código, consulta [Code Component Usage](code-component-usage.mdx).

## ¿Cuándo debo usar un componente personalizado frente a un componente de código?

La mayoría de los problemas específicos de la plataforma se pueden resolver de ambas maneras: con componentes personalizados o con el componente de código.
Al elegir cuál es el adecuado para tu situación, considera dos cosas: reutilización y complejidad.


- **Reusabilidad**: Si estás escribiendo código que podría reutilizarse en varias integraciones, deberías escribir un componente personalizado.
- **Complejidad**: Si tu código es lo suficientemente complejo como para beneficiarse de las pruebas unitarias, variables fuertemente tipadas, etc., deberías escribir un componente personalizado.

Si tu código es específico de la integración, no es muy complejo y no depende del almacén de credenciales de , un componente de código será suficiente para ti.

## La biblioteca de componentes personalizados

La biblioteca de componentes personalizados, que llamamos [Spectral](spectral/custom-component-library.mdx), proporciona definiciones de tipo TypeScript para su uso en el desarrollo de componentes personalizados.
La documentación sobre Spectral está disponible [aquí](spectral/custom-component-library.mdx).
