### Integración del Ejecutor Cambiando a NodeJS 18.x

**Anuncio de Plataforma**: Nos complace anunciar que estaremos realizando la transición de nuestro ejecutor de integraciones de NodeJS 14.x a 18.x a partir del lunes 21 de agosto.
Ejecutarse en la última versión LTS de NodeJS tiene enormes ventajas, incluidas mejoras de rendimiento, soporte continuo y parches de seguridad, y una [API de fetch](https://developer.mozilla.org/es/docs/Web/API/fetch) incorporada.

Este cambio no requerirá ninguna acción de tu parte.
Tus flujos y componentes se trasladarán sin problemas a su nuevo entorno.
Te informaremos una vez que se complete la actualización, para que puedas aprovechar cualquier nueva característica que ofrezca NodeJS 18.x.

</ChangeLogEntry>

### Diseñador Integrado

Tus clientes pueden querer construir integraciones por sí mismos entre tu producto y otras aplicaciones y servicios que utilizan.
El diseñador integrado permite a tus clientes iniciar sesión en tu aplicación y construir integraciones usando el diseñador de integraciones.
Puedes proporcionar a tus clientes acceso a componentes privados que has construido, y puedes ofrecerles a tus clientes "plantillas" que pueden usar como punto de partida para construir sus propias integraciones.

<Vimeo video="852820250" title="Anuncio de Embedded Designer" />

Lee más sobre el diseñador integrado en la página de documentación [Incrustación del Diseñador de Integración](embedded-designer.mdx).

Si estás interesado en utilizar el diseñador integrado, por favor [contacta al soporte](mailto:support@.io) para discutir la activación del diseñador integrado para tu organización.

### Nuevo SDK Incrustado

Junto con el diseñador integrado viene un nuevo SDK incrustado.
Si actualmente estás utilizando [@-io/marketplace](https://www.npmjs.com/package/@-io/marketplace) para el mercado incrustado, cámbialo por el nuevo paquete de NodeJS [@-io/embedded](https://www.npmjs.com/package/@-io/embedded) para comenzar a utilizar las funciones del diseñador integrado.
`@-io/marketplace` seguirá funcionando, pero `@-io/embedded` es un conjunto de características y nuevas características y mejoras solo se agregarán al nuevo SDK incrustado.

### Filtrado Avanzado en el Marketplace Integrado

Ahora puedes usar operadores lógicos como `and`, `or`, `startsWith`, `notEqual` y más para filtrar las integraciones en tu marketplace integrado.
Por ejemplo, si deseas mostrar todas las integraciones que tengan una categoría "ERP" y una etiqueta "pago", y también deseas que se muestren tus integraciones de Dropbox y Slack, un filtro podría ser así:

```ts
[
  BooleanOperator.or,
  [
    BooleanOperator.and,
    [TermOperator.equal, "category", "ERP"],
    [TermOperator.in, "labels", "paid"],
  ],
  [TermOperator.equal, "name", "Dropbox"],
  [TermOperator.equal, "name", "Slack"],
];
```

Lee más sobre los filtros avanzados en la [documentación](embedding-marketplace.mdx#filtros-de-integración-avanzados).

</ChangeLogEntry>

### Configuración a Nivel de Usuario

Nos enorgullece presentar una nueva opción de configuración de instancia: **Configuración a Nivel de Usuario** (ULC).

**¿Qué es ULC**?
ULC es útil cuando varios usuarios de un único cliente necesitan una integración.
Permite configurar una sola instancia de una integración para un cliente, pero recopilar configuración de múltiples usuarios y ejecutarla utilizando la configuración específica del usuario.

**¿Por qué usar ULC**?
ULC es útil si tu integración requiere variables de configuración y credenciales específicas del usuario.
Por ejemplo, supongamos que tu aplicación necesita escribir datos en las carpetas privadas de Dropbox de varios usuarios.
Con ULC, puedes recopilar información de conexión para varios usuarios dentro de un cliente e integrarte con cada una de sus cuentas individuales de Dropbox.

**¿Cómo funciona ULC**?
A grandes rasgos, se implementa una sola instancia de una integración para un cliente y se configura con algunas variables de configuración específicas del cliente.
Luego, los usuarios individuales dentro del cliente pasan por un asistente de configuración de ULC y proporcionan credenciales específicas del usuario y variables de configuración.
Cuando se ejecuta la instancia, esta utiliza la configuración específica del usuario según algunas reglas que estableces.

Lee más sobre ULC en nuestra [documentación](user-level-configuration.mdx).

</ChangeLogEntry>

### Autenticación más simple y flexible

Hemos renovado la forma en que los componentes se conectan a aplicaciones y servicios de terceros.
El nuevo concepto se llama **conexiones**, y hacen que la autenticación sea _más simple_, más _flexible_ y _más fácil de mantener_.

Estaremos actualizando nuestros componentes integrados para usar conexiones en las próximas semanas, y las credenciales eventualmente serán eliminadas gradualmente a favor de las conexiones (¡pero no te preocupes, las credenciales no serán eliminadas inmediatamente!).

Para obtener una descripción completa de cómo las conexiones mejoran el desarrollo de integraciones y el soporte, y la implementación autoasistida por parte del cliente, consulta nuestro <InternalLink href="/blog/simpler-more-flexible-authentication/">anuncio en el blog</InternalLink>.
Aquí tienes un resumen rápido:

- Los desarrolladores de componentes tienen _más flexibilidad_ al declarar qué información necesitan sus componentes para conectarse a un tercero.
  Pueden definir [conexiones personalizadas](custom-components/writing-custom-components.mdx#adding-connections) que incluyan cualquier cantidad de campos, como nombre de usuario, contraseña, clave API, ID de inquilino, URL de punto de conexión u otros campos que sean únicos para el servicio con el que se están integrando.
- El flujo de [OAuth 2.0](configuration-wizard.mdx#oauth-20-connection-config-variables) se volvió mucho _más simple_ y limpio tanto para los constructores de integraciones como para los clientes que implementan la integración: los clientes solo ven un solo botón para hacer clic cuando necesitan autenticarse con OAuth.
- La autorización se volvió _más simple_ en general: las conexiones están dentro del diseñador de integraciones o de una instancia implementada.
  No necesitas crear credenciales desde las páginas de configuración de la organización o del cliente, ni manejar diferentes tipos de credenciales.
  Los componentes saben con qué conexiones son compatibles y solo se pueden vincular con esas variables de configuración de conexión.
- Las conexiones son _más fáciles de mantener_.
  Ahora puedes configurar monitores de alerta para [notificarte](monitoring-and-alerting.mdx#alerting-on-connection-errors) cuando las conexiones (OAuth u otras) caduquen o no puedan autenticarse en una integración.

</ChangeLogEntry>

### Escribe tus Propios Disparadores

La gran mayoría de las integraciones se activan de una de dos formas: se ejecutan según un horario (por ejemplo, "A los 15 minutos de cada hora") o son invocadas por una solicitud HTTP a un [webhook](integration-triggers.mdx#webhook-triggers).

Sin embargo, no todas las aplicaciones y servicios con los que te integras son iguales, y algunas requieren funcionalidades o validaciones adicionales.
Por ejemplo, los mensajes salientes de Salesforce [outbound messages](https://developer.salesforce.com/docs/atlas.en-us.api.meta/api/sforce_api_om_outboundmessaging_understanding.htm) (webhooks) requieren una respuesta de confirmación especial en formato XML (ACK) a una solicitud de webhook, y el Servicio de Notificación Simple (SNS) de Amazon requiere que las integraciones envíen una solicitud HTTP POST a AWS para [confirmar una suscripción SNS](https://docs.aws.amazon.com/sns/latest/dg/SendMessageToHttp.prepare.html).

Teniendo en cuenta esas consideraciones, hemos ampliado nuestro [SDK de componentes personalizados](spectral/custom-component-library.mdx) para permitirte escribir tus propios disparadores para tus componentes.
Tus disparadores pueden manejar cosas como:

- Responder a solicitudes de webhook con respuestas personalizadas
- Validar los encabezados del webhook y los datos del payload
- Transformar y procesar formatos de datos XML, CSV u otros formatos propietarios para que el resto de tu integración pueda referenciar fácilmente los datos que llegan

Si has [escrito tus propias acciones](custom-components/writing-custom-components.mdx#writing-actions), escribir un disparador se sentirá muy familiar.
Consulta nuestra documentación sobre [escritura de disparadores](custom-components/writing-custom-components.mdx#writing-triggers) para comenzar, o echa un vistazo a nuestra [guía rápida](quickstarts/writing-a-trigger.mdx) que detalla cómo escribimos el disparador de Salesforce.

### Puntos finales de Webhook Configurables

Ahora tienes más control sobre cómo se configuran los puntos finales de webhook para las instancias desplegadas.
Ya tenías la opción de crear un punto final de webhook para cada flujo de cada instancia desplegada (**Específico de la Instancia y del Flujo**).
Ahora puedes seleccionar otras dos opciones de configuración:

- **Específico de la Instancia**: Crea un único punto final de webhook para cada instancia.
  Identifica qué flujos de las instancias deben ejecutarse en función de los datos en la solicitud de webhook.
- **Compartido**: Crea un único webhook que es compartido por todos los clientes que tienen una integración particular.
  Rutea la solicitud de webhook a un flujo en la instancia de un cliente específico en función de los datos en la solicitud de webhook.

Ambas opciones de configuración adicionales te permiten enrutar las solicitudes de webhook a un cliente y flujo particular en función de los datos que llegan al webhook.
Si los datos que llegan necesitan procesamiento adicional, o si necesitas buscar el nombre de un flujo o el ID del cliente, puedes asignar uno de los flujos de tu integración para que sea un **Flujo de Preprocesamiento** - un flujo que se ejecuta cuando se invoca un webhook y ayuda a asegurar que la solicitud llegue al lugar correcto.

Para obtener más información, consulta nuestros [documentos sobre webhook](/integration-triggers.mdx#webhook-endpoint-configuration) y [Quickstart de Webhook Compartido](quickstarts/using-shared-webhooks-and-preprocess-flows.mdx).

### Integraciones Multiflujo

¡Ahora proporciona soporte completo para integraciones de múltiples flujos!

Las integraciones ahora pueden incluir múltiples **flujos**. (Un flujo es un disparador y una serie de pasos.) Esto le permite proporcionar a sus clientes una integración de terceros compleja que realiza múltiples tareas relacionadas, pero se empaqueta y despliega como una sola integración.

Por ejemplo, podría integrarse con un ERP que envía una variedad de datos a través de webhooks a su aplicación (un webhook cuando se actualiza el inventario, un webhook cuando se actualiza la información del cliente, y así sucesivamente). En lugar de construir integraciones con ramificaciones complejas o ensamblar múltiples integraciones, ahora puede crear una sola integración con flujos que manejen cada tipo de carga útil de webhook. Crearía un flujo para manejar las actualizaciones de inventario, otro flujo para manejar las actualizaciones de clientes, y desplegaría todos esos flujos juntos como una sola instancia para un cliente.

Cada flujo tiene su propio disparador (por lo que obtiene su propia URL de webhook), y los flujos se prueban y ejecutan de forma independiente entre sí.

Cuando los clientes o los equipos orientados al cliente implementan una integración de múltiples flujos, configuran y despliegan todos los flujos a la vez usando una sola pantalla de configuración.

Las integraciones existentes seguirán funcionando como hasta ahora. Tenga en cuenta lo siguiente:

1. El desplegable **Ejecución de prueba seleccionada** se ha movido al selector de referencia de entrada.
   Al configurar una entrada para un paso, puede seleccionar para qué ejecución de prueba desea previsualizar las salidas dentro de la pestaña **Referencia**.

   <Screenshot
     filename="changelog/selected-test-run.png"
     maxWidth="400px"
     alt="Seleccione la ejecución de la prueba en la aplicación"
   />

2. Prism, la interfaz de línea de comandos, se ha actualizado a la versión 3.0.0 para tener en cuenta este cambio.
   Para instalar la última versión de Prism, ejecute

   ```bash
   npm install --global @-io/prism
   ```

3. Se ha actualizado el YAML que define las integraciones.
   Consulta el artículo de documentación sobre [Integraciones](integrations.mdx#yaml-definition).

Echa un vistazo a nuestra página sobre [construcción de integraciones](building-integrations.mdx#flows-in-integrations) para obtener más información, y nuestra [Guía rápida de múltiples flujos](quickstarts/building-an-integration-with-multiple-flows.mdx) para ver un ejemplo de cómo se pueden usar múltiples flujos en una integración.

### Disparadores en Tiempo de Implementación

Ahora puedes configurar flujos de integración con disparadores que se invocan cuando una instancia se implementa en un cliente. Esto es útil si tienes un conjunto de tareas de "inicialización" que deben completarse _una vez_ para configurar la instancia de un cliente.

Un flujo en tiempo de implementación podría habilitar funciones en una aplicación de terceros, configurar usuarios o permisos de terceros, crear una estructura de directorios en un sistema de almacenamiento de archivos, o incluso configurar webhooks en una aplicación de terceros para que apunten a los otros flujos de la instancia.

Consulta nuestra documentación sobre [disparadores de implementación](integration-triggers.mdx#instance-deploy-trigger) para obtener más información.

### Mejora de la Versionamiento para Componentes, Integraciones e Instancias

Se ha mejorado el versionamiento para componentes, integraciones e instancias para brindarte un control más detallado sobre exactamente qué código se implementa en los clientes.

Los **componentes** ahora se les asigna una versión entera que se incrementa cada vez que se publica el componente.
Si un componente personalizado está en "versión 3" y publicas una nueva definición del componente, esa nueva definición obtiene la "versión 4".
Esto te permite actualizar o extender componentes sin afectar involuntariamente a las integraciones existentes que los utilizan, garantizando que tus integraciones permanezcan estables.
Los creadores de integraciones pueden luego actualizar las versiones de los componentes utilizados en sus integraciones, o revertir a versiones anteriores cuando lo deseen, y serán notificados cuando haya nuevas versiones de componentes disponibles.

Lee más sobre [Versionamiento de Componentes](components.mdx#component-versioning) y [Elección de Versiones de Componentes en Integraciones](building-integrations.mdx#choosing-component-versions).

El versionamiento de **integraciones** ha sido mejorado, brindándote más control sobre qué versiones de integraciones implementas en los clientes.
Cuando publicas nuevos cambios en una integración, similar a los componentes, se le asigna un nuevo número de versión a tu integración.
Entonces, cuando implementas una instancia en un cliente, puedes elegir qué versión de la integración utilizar.
Eso significa que puedes tener algunos clientes en la versión 1 y otros en la versión 2 según sea necesario, dándote control sobre qué clientes tienen qué, y permitiéndote probar una nueva versión de integración con un pequeño subconjunto de tu base de clientes antes de implementarla ampliamente.

Revertir una implementación de instancia es muy fácil: si implementas una nueva versión de una integración a un cliente y algo parece incorrecto, puedes revertir fácilmente tu instancia a una versión conocida y funcional de la integración con solo un par de clics.

Como siempre, la actualización de las instancias de los clientes se puede automatizar, por lo que no necesitas implementar manualmente una nueva versión de una integración a cada cliente.

Lee más sobre [Publicación de una Integración](integrations.mdx#publishing-an-integration) y [Especificación de la Versión de Integración en Instancias](instances.mdx#setting-integration-version-for-an-instance).

</ChangeLogEntry>

### Reintentar y Repetir

Las organizaciones con un plan profesional o empresarial ahora pueden configurar instancias para [reintentar automáticamente](building-integrations.mdx#integration-retry-configuration) si una ejecución falla.
Puedes controlar cuántas veces una instancia intenta ejecutarse con la misma entrada y cuánto tiempo debe esperar entre intentos fallidos.
Si tienes una integración que depende de una API de terceros poco confiable, por ejemplo, esto minimiza las interrupciones tanto para tus clientes como para tu equipo.

También puedes [repetir](instances.mdx#instance-execution-retry-and-replay) -reintentar manualmente- una ejecución específica fallida de una instancia.

### Invocación de Instancias de Forma Sincrónica

Ahora puedes elegir invocar tus instancias [de forma síncrona o asíncrona](integration-triggers.mdx#synchronous-and-asynchronous-integrations).
Cuando invocas una instancia de forma síncrona, tu solicitud permanece abierta hasta que la instancia se completa y los resultados de la ejecución de la instancia se devuelven como una respuesta HTTP.
